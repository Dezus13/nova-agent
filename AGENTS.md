# Правила работы AI в проекте Nova Agent

## Основной порядок

Nova Agent разрабатывается через спецификации.

Перед выполнением любой задачи AI обязан:

1. Прочитать `docs/бизнес-контекст.md`.
2. Прочитать `docs/глобальная-спецификация.md`.
3. Определить затронутые спецификации.
4. Прочитать связанные спецификации.
5. Проверить `docs/принципы-продукта.md`.
6. Проверить `docs/roadmap.md`.
7. Проверить активный план в `plans/active`.
8. Сообщить, если обязательный документ отсутствует, пустой или противоречит задаче.
9. Только после проверки документов приступать к изменениям.

## Источник истины

Приоритет источников:

1. `docs/бизнес-контекст.md`
2. `docs/глобальная-спецификация.md`
3. `docs/карта-функций.md`
4. `docs/пользовательские-сценарии.md`
5. `docs/принципы-продукта.md`
6. `docs/roadmap.md`
7. `docs/техническая-архитектура.md`
8. `docs/specs/функции/*`
9. `docs/specs/технические/*`
10. `docs/specs/ui/*`
11. `docs/decisions/*`
12. `plans/active/*`
13. Реализация

Если документ нижнего уровня противоречит документу более высокого уровня, приоритет имеет документ более высокого уровня.

`docs/decisions/*` — архитектурные Decision Records. Фиксируют решения между specs: закрывают открытые вопросы перед следующим spec. Не заменяют specs. После появления соответствующего spec — spec имеет приоритет над Decision Record в пересекающемся scope.

`docs/принципы-продукта.md` фиксирует продуктовые принципы между пользовательскими сценариями и roadmap. Документ не создаёт новые функции, роли, сущности или состояния.

`docs/specs/функции/*`, `docs/specs/технические/*` и `docs/specs/ui/*` не являются главным источником истины выше карты функций, пользовательских сценариев, принципов продукта, roadmap и технической архитектуры.

Спецификации детализируют уже утверждённые функции, но не создают новые продуктовые направления сами по себе.

## Перед изменением

Перед любым изменением AI обязан указать:

- что меняется;
- зачем это меняется;
- какие документы затрагиваются;
- какие риски возникают;
- нужно ли обновить спецификации;
- нужно ли обновить план.

## Запрещено

AI запрещено:

- писать код без связанной спецификации;
- реализовывать функции, отсутствующие в спецификациях;
- изменять архитектуру без обновления связанных спецификаций;
- создавать временные файлы в репозитории;
- создавать дублирующие документы;
- создавать папки, не предусмотренные текущей задачей или спецификацией;
- удалять документы без указания причины;
- оставлять неиспользуемые файлы после завершения задачи;
- хранить устаревшие версии документов как отдельные копии;
- принимать архитектурные решения без проверки влияния на спецификации, план и существующую структуру проекта.

## Безопасность и продуктовые заявления

AI запрещено:

- добавлять или коммитить реальные ключи, пароли, `.env.local`, service role key или другие secrets;
- использовать service role key во frontend repository;
- утверждать, что Supabase, auth или persistence реализованы, если отсутствует runtime integration;
- начинать runtime Supabase integration без отдельного active plan, аудита и явного разрешения;
- делать fake AI/OpenAI claims или обещать real AI behavior без реализованной и задокументированной интеграции;
- описывать browser speech-to-text demo как полноценного AI voice agent.

Supabase readiness files, `.env.example` и handoff-документация означают только готовность к будущему подключению. Они не означают runtime persistence.

## Спецификации

Если функция отсутствует в спецификациях, функция не подлежит реализации.

Порядок работы с функциями:

1. Создать или обновить спецификацию.
2. Создать или обновить активный план.
3. Выполнить реализацию.
4. Проверить результат.
5. Обновить changelog.
6. Предложить commit.

## Планы

Каждая задача должна иметь связь с активным планом в `plans/active`.

Если активный план отсутствует, AI обязан сообщить об этом до начала изменений.

После завершения этапа AI обязан:

- обновить связанный активный план;
- перенести завершенный план в `plans/completed`, если этап полностью закрыт;
- сообщить, если план не был обновлен из-за его отсутствия.

Активные планы могут оставаться открытыми только как актуальные roadmap/umbrella plans или явно незавершенные этапы. Завершенные, устаревшие или замененные планы должны переноситься в `plans/completed` с evidence и результатами проверок.

## Git

Перед commit AI обязан проверить:

- `git status`;
- `git diff` для измененных файлов.

AI не должен выполнять commit без явного запроса пользователя.

После завершения задачи AI обязан предложить текст commit message.

AI не должен откатывать изменения пользователя без явного запроса пользователя.

## Changelog

После завершения задачи AI обязан обновить файл changelog.

Формат файла changelog:

- путь: `changelog/YYYY-MM-DD.md`;
- одна дата соответствует одному файлу;
- новые записи добавляются в существующий файл за дату выполнения задачи;
- запись должна перечислять измененные документы или части проекта.

Changelog обязателен для meaningful docs, product, plan, handoff and submission-readiness changes, даже если application code не менялся.

## Последовательность проекта

Обязательная последовательность работы:

1. Стратегия.
2. Спецификации.
3. Планы.
4. Реализация.
5. Проверка.
6. Changelog.
7. Commit.
