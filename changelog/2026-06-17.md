# Changelog 2026-06-17

## Документация

- Внесён TS05 Amendment (AMENDMENT 01–06) в `docs/specs/технические/05-supabase-schema-and-rls.md`: закрыты архитектурные пробелы TS05 перед переходом к TS06 Content Administration Operations.

### AMENDMENT 01 — Warning/Restriction/Applicability Condition entity immutability

- Зафиксировано: Warning, Restriction и Applicability Condition entity rows становятся immutable после первого использования в published Scenario Version.
- Добавлены правила entity-level immutability в разделы `warnings`, `restrictions`, `applicability_conditions`.
- Добавлены три новые строки в Immutability Rules (entity rows referenced by published context).
- Добавлено правило replacement: изменение Warning/Restriction/AC после published использования требует нового entity и новой Scenario Version.
- Добавлено решение перед SQL: механизм enforcement через trigger или RLS-based check.

### AMENDMENT 02 — Publication State Transition Graph

- Добавлен новый раздел `## Publication State Transition Rules` с полным утверждённым графом переходов.
- Разрешённые переходы: draft → published, published → deprecated, published → superseded, deprecated → superseded.
- Запрещённые переходы: все прочие, включая self-transitions и любой переход из superseded.
- superseded зафиксирован как финальное состояние.
- Добавлены три новые правила в `scenario_versions`.
- Добавлено правило replacement: publication_state не нарушает утверждённый граф.
- Добавлены правила append-only: content_publication_events append-only, reverse event невозможен.

### AMENDMENT 03 — Scenario Version Access Model

- Зафиксированы два read pattern для User: Pattern A (publication_state = 'published' без ограничений), Pattern B (deprecated/superseded — только при наличии Action Plan пользователя).
- Обновлены RLS-границы для User: заменена строка "читать published Scenario Version" на явные Pattern A и Pattern B.
- Добавлены два новых запрещённых RLS-паттерна: создание Action Plan для non-published; чтение deprecated/superseded без Action Plan.
- Добавлен Action Plan creation gate в правила `action_plans`.
- Добавлен partial unique constraint: один draft Scenario Version per stable Scenario identity.
- Добавлены решения перед SQL: реализация RLS Pattern B и physical enforcement partial unique.

### AMENDMENT 04 — Source Revision Boundary

- Зафиксировано: Source Revision mutable до первого publication reference; после reference — immutable.
- Добавлены четыре новые правила в `source_revisions`, включая явное разрешение mutable draft и запрет editorial правок к published Source Revision.
- Добавлено уточнение в Immutability Rules: unreferenced Source Revision остаётся mutable до publication reference.

### AMENDMENT 05 — Content Publication Events

- Полностью заменён раздел `content_publication_events`: добавлен conceptual payload (9 полей с описаниями), расширены правила, добавлен список запрещённых данных.
- Добавлен новый enum в Enum Model: type content publication event.
- Добавлены три новые FK в Foreign Key Model: content_publication_events → scenarios, → scenario_versions (affected), → scenario_versions (successor, nullable).
- Добавлено решение перед SQL: retention policy (payload теперь определён, retention остаётся открытым).

### AMENDMENT 06 — Draft Lifecycle Completion

- Добавлен новый раздел `## Draft Scenario Version Deletion` с полными правилами физического удаления draft Scenario Version.
- Зафиксирован cascade delete: удаляются все versioned context link rows; underlying content entities не затрагиваются.
- Зафиксирован запрет физического удаления non-draft Scenario Version при наличии User-Owned Data зависимостей.
- Добавлены два новых запрещённых RLS-паттерна.
- Добавлены решения перед SQL: механизм cascade delete и race condition handling.

### AMENDMENT 07 — TS06 Content Administration Operations

- Создан `docs/specs/технические/06-content-administration-operations.md`.
- Зафиксирована операционная модель Content Administration: разрешённые и запрещённые операции для 11 Content Model entities.
- Зафиксированы правила publication workflow: publish, deprecate, supersede.
- Зафиксированы правила draft lifecycle и draft deletion (cascade list из TS05 Amendment 06).
- Зафиксирована Source Revision mutable boundary: draft reference ≠ publication reference.
- Явно закрыт открытый вопрос TS04: review/approval workflow не входит в MVP; любой content_admin может publish, deprecate, supersede.
- Зафиксированы entity immutability rules: Warning/Restriction/AC, Source Revision, Versioned Content Context.
- Зафиксировано: Content Admin не имеет доступа к User-Owned Data и агрегатам; content audit = content_publication_events.
- Зафиксированы паттерны замены immutable entities (Warning, Restriction, AC).
- Перечислены инварианты, риски, решения перед SQL и критерии приёмки.
