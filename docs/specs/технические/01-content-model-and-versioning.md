# Technical Spec 01: Content Model & Versioning

## Назначение документа

Документ фиксирует техническую модель контента и версионности Nova Agent перед переходом к Logical Data Model.

Цель документа — определить:

- границу между Content Model и User-Owned Data;
- роль Scenario Version;
- правила версионности контента;
- правила публикации и обновления контента;
- правила сохранения исторического контекста активных пользовательских планов;
- обязательные связи пользовательских сущностей с конкретной версией сценария;
- инварианты, которые не должны нарушаться в следующих technical specs.

Документ не описывает Supabase, SQL, таблицы, API, RLS, UI, код, инфраструктуру или конкретную реализацию хранения данных.

## Источники и приоритет документов

Technical Spec 01 основан на документах верхнего уровня и функциональных спецификациях:

Этот документ не создаёт новые продуктовые функции. Он технически уточняет уже утверждённые функциональные границы MVP.

## Границы документа

Входит в документ:

- техническое разделение контентных и пользовательских сущностей;
- правила immutable-публикации сценариев;
- роль версии сценария в активных планах;
- правила связи прогресса, истории, пользовательских открытых вопросов и заметок с версией сценария;
- правила сохранения контекста при изменении контента;
- технические запреты, которые должны соблюдаться в следующих specs.

Не входит в документ:

- выбор базы данных;
- модель хранения;
- структура запросов;
- авторизация;
- интерфейс администратора;
- интерфейс пользователя;
- реализация публикации;
- миграции активных планов между версиями;
- хранение пользовательских документов;
- автоматическое выполнение действий от имени пользователя.

## Основные термины

### Content Model

Content Model — набор администрируемых сущностей, которые описывают структуру справочной, организационной и навигационной помощи Nova Agent.

Content Model отвечает на вопросы:

- какая жизненная ситуация описана;
- какой сценарий применим;
- какие шаги входят в сценарий;
- какие документы или данные могут понадобиться;
- какие источники нужно проверить;
- какие ограничения, предупреждения и Template Open Questions должны быть показаны.

Content Model не хранит действия, статусы, заметки, историю или решения конкретного пользователя.

### User-Owned Data

User-Owned Data — данные, которые возникают из действий конкретного пользователя внутри Nova Agent.

User-Owned Data отвечает на вопросы:

- какой сценарий пользователь начал проходить;
- какой прогресс пользователь отметил;
- какие события произошли внутри плана;
- какие вопросы пользователь зафиксировал для внешней проверки;
- какие заметки пользователь оставил к событию или контексту;
- какие источники пользователь отметил как проверенные.

User-Owned Data не является частью Content Model и не управляется администратором контента.

### Scenario

Scenario — администрируемая контентная сущность, описывающая путь пользователя в рамках одной или нескольких жизненных ситуаций.

Scenario является логическим контейнером для версий. Пользовательский план не должен создаваться на основе live-состояния Scenario без указания конкретной версии.

### Scenario Version

Scenario Version — опубликованная версия сценария, фиксирующая конкретный состав сценария на момент публикации.

Scenario Version является центральной сущностью модели версионности. Именно она связывает администрируемый контент с пользовательским планом действий.

### Action Plan

Action Plan — пользовательский экземпляр прохождения выбранного сценария.

Action Plan не является контентной сущностью, не является самостоятельным каталогом и не создаёт новый сценарий. Он всегда связан с конкретной Scenario Version.

## Технические инварианты

Следующие правила являются обязательными для всех последующих technical specs:

- Action Plan всегда связан с конкретной Scenario Version.
- Action Plan не может быть создан только на основе live Scenario без версии.
- Активный план не обновляется автоматически после публикации новой версии сценария.
- Опубликованная версия сценария является immutable.
- Пользовательские данные не являются частью Content Model.
- Администратор контента не управляет пользовательскими данными.
- Обновление контента не переписывает пользовательскую историю.
- История не переписывается после обновления контента.
- User Note не является источником информации.
- User Open Question не является консультацией, профессиональным ответом или официальным решением.
- Document Requirement не является пользовательским документом.
- Data Requirement не является заполненным пользовательским значением.
- Source не является пользовательской заметкой.
- Checked Source Mark не является изменением Source.
- Progress не является официальным статусом внешнего процесса.
- History Event не является официальным журналом взаимодействия с органом, учреждением или специалистом.
- Template Open Question и User Open Question являются разными сущностями.
- Изменение Content Model не должно автоматически изменять User-Owned Data.
- Published content, используемый активными планами или историей, не должен удаляться так, чтобы был потерян исторический контекст.

## Content Model Entities

К Content Model относятся:

- Life Situation;
- Scenario;
- Scenario Version;
- Step Template;
- Document Requirement;
- Data Requirement;
- Source;
- Template Open Question;
- Restriction;
- Warning;
- Applicability Condition;
- Step Order;
- Step Dependency;
- связи между Life Situation и Scenario;
- связи между Scenario Version и шагами, документами, данными, источниками, ограничениями, предупреждениями и Template Open Questions;
- metadata публикации контента.

Content Model управляется администратором контента в рамках утверждённых спецификаций.

Content Model не включает:

- пользовательский план;
- пользовательский прогресс;
- пользовательскую историю;
- пользовательские заметки;
- пользовательские открытые вопросы;
- пользовательские отметки о проверке источника;
- пользовательские документы;
- заполненные пользовательские значения;
- официальные ответы внешних сторон.

## User-Owned Data Entities

К User-Owned Data относятся:

- User;
- Action Plan;
- Plan Lifecycle State;
- Progress;
- History Event;
- User Open Question;
- User Note;
- Checked Source Mark;
- пользовательское действие, которое создаёт или изменяет состояние внутри плана;
- пользовательские признаки скрытия, удаления или архивирования, если они будут утверждены следующими specs.

User-Owned Data принадлежит пользователю и существует внутри пользовательского прохождения.

User-Owned Data может ссылаться на stable content identity и Scenario Version, но не должна изменять Content Model.

Если смысл пользовательского состояния зависит от контента, ссылка должна идти через Scenario Version или versioned content context.

User-Owned Data не должна обходить Scenario Version прямой ссылкой на mutable live content.

## Versioned Entities

В MVP версионирование должно защищать опубликованный контент и исторический контекст активных планов.

К Versioned Entities относятся:

- Scenario Version;
- состав сценария внутри Scenario Version;
- Step Template внутри опубликованной версии сценария;
- порядок шагов внутри опубликованной версии сценария;
- зависимости шагов внутри опубликованной версии сценария;
- Document Requirement внутри опубликованной версии сценария;
- Data Requirement внутри опубликованной версии сценария;
- Source reference или source revision, использованный в опубликованной версии сценария;
- Template Open Question внутри опубликованной версии сценария;
- Restriction внутри опубликованной версии сценария;
- Warning внутри опубликованной версии сценария;
- Applicability Condition внутри опубликованной версии сценария;
- связи Scenario Version с Life Situation, если эта связь влияет на пользовательский контекст плана.

Versioned Entity не означает, что каждая сущность обязана иметь самостоятельную независимую версию. Для MVP допустимо, что дочерние элементы наследуют immutable-контекст Scenario Version, если их состояние зафиксировано внутри опубликованной версии сценария.

Source требует особого внимания, потому что один источник может использоваться в разных сценариях и меняться независимо от них. Следующие technical specs должны выбрать, фиксируется ли источник через source revision или через snapshot ключевых source-данных в Scenario Version.

## Non-Versioned Entities

К Non-Versioned Entities относятся пользовательские сущности, которые не являются версиями контента:

- User;
- Action Plan;
- Plan Lifecycle State;
- Progress;
- History Event;
- User Open Question;
- User Note;
- Checked Source Mark.

Эти сущности не получают content-version сами по себе. Вместо этого они должны сохранять связь с конкретной Scenario Version и, когда нужно, с конкретным элементом опубликованной версии сценария.

User-Owned Data может иметь собственный lifecycle, audit trail или историю изменений, но это не является версионностью Content Model.

## Scenario Version как центральная сущность модели

Scenario Version является технической границей между администрируемым контентом и пользовательским прохождением.

Scenario Version фиксирует:

- к какому Scenario относится версия;
- какой состав контента был опубликован;
- какие шаги входили в сценарий;
- в каком порядке были расположены шаги;
- какие документы и данные были связаны со сценарием или шагами;
- какие источники были связаны со сценарием, шагами, документами, данными или вопросами;
- какие ограничения и предупреждения были актуальны;
- какие Template Open Questions были частью опубликованного сценария;
- какие Life Situations были связаны с версией, если это влияет на пользовательский выбор.

Action Plan создаётся на основе Scenario Version, а не на основе изменяемого live Scenario.

Если позже публикуется новая Scenario Version, существующий Action Plan остаётся связанным с исходной Scenario Version.

Переход активного плана на новую Scenario Version не входит в MVP как автоматическое поведение и требует отдельного продуктового решения.

## Правила публикации контента

Публикация контента означает создание новой immutable Scenario Version.

При публикации должны быть зафиксированы:

- состав сценария;
- порядок шагов;
- зависимости шагов;
- связанные Document Requirements;
- связанные Data Requirements;
- связанные Sources или их исторически значимый контекст;
- Template Open Questions;
- Restrictions;
- Warnings;
- Applicability Conditions;
- связи с Life Situations, если они нужны для контекста плана.

После публикации Scenario Version не должна изменяться задним числом.

Published Scenario Version никогда не мутируется.

Если требуется изменить опубликованный сценарий, должна быть подготовлена новая Scenario Version.

Replacement или deprecation допустимы только через новую Scenario Version, новую revision или superseding-связь, которая сохраняет исторический контекст уже опубликованной версии.

Нельзя заменять дочерний элемент внутри уже опубликованной Scenario Version.

## Правила обновления контента

Администратор контента может обновлять Content Model только как контент, а не как пользовательское прохождение.

Обновление контента может включать:

- уточнение сценария;
- добавление, изменение или удаление шага в новой версии;
- изменение порядка шагов в новой версии;
- изменение Document Requirements;
- изменение Data Requirements;
- обновление Source;
- изменение Restrictions;
- изменение Warnings;
- изменение Template Open Questions;
- изменение Applicability Conditions.

Обновление контента не должно:

- менять активные Action Plans автоматически;
- сбрасывать Progress;
- переписывать History Events;
- удалять User Open Questions;
- редактировать User Notes;
- менять смысл Checked Source Marks;
- скрывать, что пользователь действовал в контексте предыдущей версии сценария.

Активный план может оставаться связанным с предыдущей Scenario Version. Это нормальное состояние продукта.

## Правила сохранения исторического контекста

Исторический контекст нужен, чтобы пользователь мог понять, что происходило внутри плана на момент действия.

Исторический контекст должен сохраняться для:

- Action Plan;
- Progress;
- History Event;
- User Open Question;
- User Note;
- Checked Source Mark.

Минимальное правило: каждая пользовательская сущность, смысл которой зависит от контента сценария, должна быть связана с Scenario Version напрямую или через Action Plan и History Event.

Для History Event должен сохраняться контекст, достаточный для восстановления смысла события после обновления контента.

Следующие technical specs должны определить точный минимальный event payload, но этот payload не должен превращать историю в хранилище документов, официальных ответов или произвольных персональных данных.

## Action Plan и Scenario Version

Action Plan всегда связан с конкретной Scenario Version.

Эта связь нужна, чтобы:

- сохранить структуру сценария, на которой пользователь начал прохождение;
- не сбросить Progress после обновления контента;
- не переписать History;
- сохранить смысл User Notes;
- сохранить связь User Open Questions с исходным контекстом;
- сохранить смысл Checked Source Marks;
- отделить администрирование контента от пользовательского процесса.

Action Plan не должен автоматически переходить на новую Scenario Version после публикации новой версии сценария.

Если в будущем будет добавлен переход плана на новую версию, он должен быть описан отдельной спецификацией и не должен переписывать историю старой версии.

## Progress и Scenario Version

Progress принадлежит Action Plan.

Progress отражает состояние пользователя по шагам плана, созданного на основе конкретной Scenario Version.

Progress должен быть связан:

- с Action Plan;
- с конкретной Scenario Version через Action Plan;
- с конкретным шагом опубликованной версии сценария, если статус относится к шагу.

Progress не должен ссылаться только на live Step Template, потому что live-шаг может измениться после публикации новой версии сценария.

Обновление контента не должно менять Progress активного плана.

## History и Scenario Version

History принадлежит Action Plan.

History Event фиксирует событие в пользовательском плане в контексте конкретной Scenario Version.

History Event должен сохранять связь:

- с Action Plan;
- со Scenario Version;
- с релевантным шагом опубликованной версии сценария, если событие связано с шагом;
- с Source или его историческим контекстом, если событие связано с проверкой источника;
- с User Open Question, если событие связано с вопросом.

History Event не обязан иметь обязательную обратную ссылку на User Note.

Если событие связано с созданием, редактированием, скрытием или удалением User Note, это должно фиксироваться как History Event без создания обязательной циклической зависимости History Event <-> User Note.

History Event не должен переписываться после обновления контента.

Если контент обновлён, старые события истории должны оставаться понятными в контексте той версии сценария, которая была актуальна на момент действия пользователя.

## User Open Question и Scenario Version

User Open Question принадлежит Action Plan и является пользовательской сущностью.

User Open Question должен сохранять связь:

- с Action Plan;
- со Scenario Version;
- с versioned Template Open Question context внутри Scenario Version, если пользовательский вопрос возник из шаблонного вопроса;
- с шагом, документом, требованием к данным, источником или направлением внешней проверки, если вопрос связан с ними.

User Open Question не является консультацией, профессиональным ответом, официальным решением или источником факта.

User Open Question не должен ссылаться на live Template Open Question.

Обновление Template Open Question в новой версии сценария не должно автоматически менять User Open Question в активном плане.

## User Note и Scenario Version

User Note принадлежит пользователю и существует внутри Action Plan.

User Note указывает на History Event. Через History Event и Action Plan заметка получает контекст Scenario Version.

History Event не обязан иметь обязательную обратную ссылку на User Note.

Создание, редактирование, скрытие или удаление User Note должно фиксироваться как History Event.

Logical Data Model не должен создавать обязательную циклическую зависимость History Event <-> User Note.

User Note может быть связана с:

- шагом внутри опубликованной версии сценария;
- User Open Question;
- Checked Source Mark;
- изменением Progress;
- другим допустимым событием истории, если оно будет утверждено следующими specs.

User Note не является:

- источником информации;
- документом;
- требованием к данным;
- открытым вопросом;
- консультацией;
- официальным фактом;
- ответом специалиста;
- произвольным дневником;
- хранилищем пользовательских файлов или структурированных персональных данных.

Администратор контента не редактирует User Notes.

Обновление контента не должно менять User Notes или контекст событий, к которым они привязаны.

## Document Requirement и пользовательские документы

Document Requirement является частью Content Model.

Document Requirement описывает, какой документ может понадобиться пользователю, где он может быть применим и где требования нужно проверить.

Document Requirement не является пользовательским документом.

MVP не хранит:

- пользовательские файлы;
- загруженные документы;
- сканы;
- вложения;
- заполненные формы;
- официальную переписку;
- подтверждения внешних сторон как проверенные продуктом факты.

Если в будущем будет добавлено хранение документов, это потребует отдельной спецификации, обновления roadmap и отдельной оценки рисков.

## Запреты на смешение сущностей

Следующие смешения запрещены:

- Scenario и Action Plan;
- Step Template и Progress;
- Source и Checked Source Mark;
- Source и User Note;
- Template Open Question и User Open Question;
- User Open Question и консультационный ответ;
- User Note и History Event;
- User Note и Source;
- Document Requirement и пользовательский файл;
- Data Requirement и заполненное пользовательское значение;
- History и Progress;
- Content administration и управление пользовательскими данными;
- план действий и универсальный task manager.

Если следующая technical spec требует нарушить одно из этих правил, сначала должны быть обновлены продуктовые спецификации более высокого уровня.

## Правила удаления и замены контента

Контент, который уже опубликован в Scenario Version или используется активными планами, не должен удаляться так, чтобы был потерян исторический контекст.

Вместо разрушительного удаления должны использоваться состояния или правила, которые сохраняют возможность понять прошлый контекст:

- deprecated;
- replaced;
- hidden from new publishing;
- not recommended for new scenarios;
- superseded by newer content.

Точный набор состояний должен быть выбран в следующих technical specs.

Главное правило: удаление или замена контента не должны переписывать прошлые планы, историю, заметки, открытые вопросы или checked-source marks.

## Что должно быть решено до Logical Data Model

Перед Logical Data Model нужно принять решения:

- какие дочерние элементы Scenario Version имеют самостоятельный versioned identity;
- какие элементы наследуют immutable-контекст Scenario Version;
- как фиксируется исторический контекст Source;
- какой минимальный context payload нужен History Event;
- должна ли User Note иметь прямую связь со Scenario Version или только через History Event и Action Plan;
- может ли одно событие истории иметь несколько User Notes;
- какие состояния публикации контента входят в MVP;
- какие правила действуют для deprecated или replaced content;
- какие пользовательские действия создают History Event;
- какие ограничения нужны для User Open Question и User Note, чтобы они не стали консультацией, ответом, дневником или хранилищем данных.

## Что должно быть вынесено в следующие technical specs

Следующие темы не решаются в этом документе:

- Logical Data Model;
- физическое хранение данных;
- права доступа и роли на уровне реализации;
- полный lifecycle User Open Question;
- полный lifecycle User Note;
- полный enum History Events;
- точный payload History Events;
- workflow админской публикации;
- правила отображения старой версии сценария пользователю;
- правила миграции активного плана на новую Scenario Version;
- хранение пользовательских документов;
- экспорт, удаление или перенос пользовательских данных.

## Acceptance Criteria

Technical Spec 01 считается готовым, если:

- явно разделены Content Model и User-Owned Data;
- перечислены Content Model Entities;
- перечислены User-Owned Data Entities;
- перечислены Versioned Entities;
- перечислены Non-Versioned Entities;
- Scenario Version описана как центральная сущность модели;
- зафиксировано, что Action Plan всегда связан с конкретной Scenario Version;
- зафиксировано, что активный план не обновляется автоматически после публикации новой Scenario Version;
- зафиксировано, что опубликованная Scenario Version immutable;
- зафиксировано, что пользовательские данные не являются частью Content Model;
- зафиксировано, что администратор контента не управляет пользовательскими данными;
- зафиксировано, что история не переписывается после обновления контента;
- зафиксировано, что User Note не является источником информации;
- зафиксировано, что User Open Question не является консультацией или ответом;
- зафиксировано, что Document Requirement не является пользовательским документом;
- описаны правила сохранения исторического контекста;
- описаны связи Action Plan, Progress, History, User Open Question и User Note с Scenario Version;
- документ не описывает Supabase, SQL, таблицы, API, RLS, UI или код;
- документ может быть использован как источник для Logical Data Model без принятия скрытых продуктовых решений.
