# 11. Пользовательские заметки

## Назначение спецификации

Документ описывает функцию "Пользовательские заметки" для MVP Nova Agent.

Спецификация фиксирует, что считается User Note, зачем заметка нужна пользователю, кому она принадлежит, где может существовать, как связана с планом действий, историей действий, шагами, открытыми вопросами, источниками, документами и данными.

Документ не создаёт новую большую продуктовую функцию. User Note уточняет уже описанное пользовательское состояние внутри плана действий и истории действий.

Документ не выбирает техническую реализацию, базу данных, API, UI, инфраструктуру или код. Эти темы должны описываться позже в отдельных technical specs после утверждения функциональных спецификаций.

## Определение User Note

User Note — короткая пользовательская запись для восстановления контекста прохождения утверждённого сценария.

User Note отвечает на вопрос: что пользователь хотел зафиксировать для себя рядом с конкретным событием прохождения.

User Note принадлежит конкретному пользователю и конкретному плану действий.

User Note не существует без плана действий.

User Note создаётся только пользовательским действием.

User Note всегда связана с событием истории.

User Note может быть связана с:

- шагом внутри плана действий;
- открытым вопросом;
- отметкой источника;
- изменением статуса.

User Note помогает пользователю восстановить, почему он сделал отметку, что хотел помнить о событии и какой контекст был важен в момент прохождения.

## Статус функции в MVP

В MVP User Note является ограниченной пользовательской записью внутри плана действий и истории действий.

User Note не является самостоятельным разделом продукта, отдельным блокнотом или универсальным хранилищем текста.

В MVP User Notes нужны для:

- восстановления контекста при возвращении к незавершённому процессу;
- пояснения пользовательского действия внутри плана;
- сохранения короткого пользовательского контекста рядом с событием истории;
- отделения личной фиксации пользователя от источников информации, открытых вопросов и официальных фактов;
- снижения риска потери логики прохождения без хранения лишних пользовательских данных.

Добавление User Note является stateful-действием и может создать план действий, если пользователь до этого только просматривал сценарий.

Если пользователь только просматривает сценарий и не совершает stateful-действий, User Note не создаётся.

## Чем заметка не является

User Note не является источником информации.

User Note не является документом.

User Note не является открытым вопросом.

User Note не является событием истории сама по себе.

User Note не является официальным фактом.

User Note не является официальным ответом органа, учреждения или специалиста.

User Note не является юридической, налоговой, медицинской, финансовой или иной профессиональной консультацией.

User Note не является подтверждением правильности документов, достаточности документов, применимости требований или результата внешнего процесса.

User Note не является блокнотом или дневником.

User Note не является произвольной задачей, подзадачей, чек-листом, дедлайном, напоминанием или workflow.

User Note не является пользовательским файлом, вложением, заполненной формой или перепиской с внешней стороной.

User Note не является структурированным персональным значением пользователя.

## Пользовательская проблема

Пользователь проходит многошаговый процесс, который может растягиваться на дни или недели.

В процессе прохождения пользователь может помнить текущее действие, но терять контекст:

- почему он изменил статус шага;
- почему отметил источник как проверенный;
- почему создал или изменил открытый вопрос;
- что было важно в момент действия;
- что нужно вспомнить при возвращении к плану после перерыва.

Без ограниченной заметки пользователь может переносить контекст во внешние заметки, чаты, таблицы или личные документы. Это снова разрывает связь между сценарием, планом, прогрессом, историей, источниками и открытыми вопросами.

User Note должна помогать восстановить контекст прохождения без превращения Nova Agent в личный блокнот, дневник, файловое хранилище, систему задач или консультационный продукт.

## Владелец и область существования

Владельцем User Note является пользователь, который создал заметку.

User Note принадлежит конкретному пользовательскому плану действий.

User Note существует только внутри пользовательского контекста прохождения утверждённого сценария.

User Note не существует:

- без пользователя;
- без плана действий;
- вне события истории;
- внутри сценария как контентной структуры;
- внутри шаблона шага;
- внутри источника информации;
- внутри документа или требования к данным;
- внутри Template Open Question.

Один и тот же сценарий может использоваться разными пользователями, но заметки одного пользователя не становятся частью сценария и не видны как продуктовый контент.

## Связь с планом действий

User Note принадлежит плану действий.

План действий является пользовательским экземпляром выбранного сценария и связывает пользователя, жизненную ситуацию, сценарий, версию сценария, шаги, документы, источники, открытые вопросы, прогресс и историю.

User Note не может быть создана без плана действий.

Если пользователь добавляет User Note при просмотре сценария без существующего плана, это действие считается stateful-действием и может создать план действий.

User Note должна сохранять связь с тем планом, в контексте которого она была создана.

User Note не меняет:

- сценарий;
- версию сценария;
- шаблон шага;
- список документов;
- источник информации;
- открытый вопрос;
- текущий прогресс.

User Note может пояснять пользовательское действие внутри плана, но не должна становиться новым действием, новой задачей или новым шагом.

User Notes сохраняются вместе с завершённым планом действий и его историей.

Завершение плана не удаляет и не скрывает пользовательские заметки автоматически.

User Notes остаются частью пользовательского контекста завершённого плана.

## Связь с историей действий

User Note всегда связана с событием истории.

История действий отвечает на вопрос: что происходило с планом действий и в каком порядке.

User Note отвечает на вопрос: что пользователь хотел запомнить рядом с событием.

User Note не является событием истории сама по себе.

Событием истории является пользовательское действие внутри плана, например:

- изменение статуса;
- отметка источника;
- создание открытого вопроса;
- изменение открытого вопроса;
- закрытие открытого вопроса пользователем;
- создание User Note;
- редактирование User Note;
- удаление или скрытие User Note.

User Note может сопровождать событие истории как часть контекста этого события.

Заметка без события истории не входит в MVP.

История не должна строиться вокруг произвольных заметок. Она строится вокруг событий плана действий.

## Связь со шагом

User Note может быть связана с шагом внутри плана действий, если заметка поясняет пользовательский контекст события, связанного с этим шагом.

Примеры допустимой связи:

- пользователь изменил статус шага и добавил короткое пояснение;
- пользователь отметил, что шаг требует проверки, и зафиксировал, что именно хотел помнить;
- пользователь отметил шаг выполненным и добавил контекст для себя.

User Note не хранится в шаблоне шага.

User Note не меняет назначение шага, порядок шагов, зависимости или содержание сценария.

User Note не превращает шаг в произвольную пользовательскую задачу.

## Связь с открытым вопросом

User Note может быть связана с User Open Question, если заметка поясняет пользовательский контекст события по вопросу.

Открытый вопрос отвечает на вопрос: что нужно проверить.

User Note отвечает на вопрос: что пользователь зафиксировал для себя рядом с событием.

User Note не заменяет открытый вопрос.

User Note не создаёт предмет внешней проверки сама по себе.

User Note не содержит официальный ответ на открытый вопрос.

User Note не закрывает открытый вопрос автоматически.

Закрытие или изменение открытого вопроса должно оставаться отдельным пользовательским действием.

Если пользователь хочет зафиксировать неопределённость, которую нужно проверить у органа, учреждения, источника или специалиста, это должно быть User Open Question, а не User Note.

Если пользователь хочет кратко пояснить, почему он создал, изменил или закрыл вопрос, это может быть User Note, связанная с событием вопроса.

## Связь с источниками, документами и данными

Источник информации отвечает на вопрос: где проверять.

User Note отвечает на вопрос: что пользователь хотел запомнить рядом с событием.

User Note не является источником информации и не должна использоваться как источник истины.

User Note может быть связана с отметкой источника, если пользователь отметил источник как проверенный и добавил короткий контекст для себя.

User Note не подтверждает:

- актуальность источника;
- официальный статус источника;
- применимость требования;
- достаточность документа;
- правильность документа;
- результат внешнего процесса.

Документ или данные в Nova Agent описываются как требования или элементы подготовки.

User Note не является документом.

User Note не является требованием к данным.

User Note не должна хранить пользовательские файлы, вложения, сканы, заполненные формы, официальную переписку или структурированные персональные значения.

Если пользователю нужно помнить, что конкретный документ или данные могут понадобиться, продукт должен использовать связь документа или требования к данным со сценарием, шагом или открытым вопросом, а не превращать User Note в хранилище документов.

## Допустимые use cases MVP

В MVP допустимы только заметки, которые помогают восстановить контекст прохождения утверждённого сценария.

Допустимые use cases:

- короткое пояснение к изменению статуса шага;
- короткое пояснение к отметке источника как проверенного;
- короткое пояснение к созданию User Open Question;
- короткое пояснение к изменению состояния User Open Question;
- короткое пояснение к закрытию User Open Question пользователем;
- короткое пояснение к событию возврата к незавершённому процессу, если такое событие входит в историю действий MVP.

Допустимые заметки должны быть связаны с событием истории и не должны создавать самостоятельную пользовательскую область вне плана действий.

## Запрещённое содержание

User Note не должна хранить:

- пользовательские файлы;
- вложения;
- сканы документов;
- фотографии документов;
- заполненные формы;
- официальную переписку;
- письма от органов, учреждений, работодателей, страховых организаций или специалистов;
- официальные ответы как подтверждённые факты продукта;
- профессиональные заключения;
- юридические, налоговые, медицинские или финансовые выводы;
- структурированные персональные значения;
- пароли, коды доступа, токены, PIN-коды или секреты;
- данные банковских карт или платёжные реквизиты;
- произвольные дневниковые записи без связи с событием плана;
- списки задач, дедлайны, приоритеты, исполнителей или workflow.

User Note не должна использоваться для обхода ограничения MVP на хранение пользовательских документов, заполненных персональных значений или официальной переписки.

## Минимальный состав данных на продуктовом уровне

User Note должна описываться на продуктовом уровне.

Минимальный состав User Note:

- связанный пользовательский план действий;
- пользователь, который создал заметку;
- связанное событие истории;
- тип связанного события: изменение статуса, отметка источника, создание вопроса, изменение вопроса, закрытие вопроса или другое допустимое событие MVP;
- связанный шаг, если событие относится к шагу;
- связанный User Open Question, если событие относится к вопросу;
- связанный источник, если событие относится к отметке источника;
- короткое содержание заметки;
- дата, время или порядок создания внутри плана действий;
- дата, время или порядок последнего пользовательского изменения, если редактирование допустимо;
- признак удаления или скрытия, если такое состояние будет выбрано в lifecycle MVP;
- указание, что заметка является пользовательской записью и не является источником, официальным фактом или консультацией.

Эта спецификация не определяет технические поля, формат хранения, API, UI или структуру базы данных.

## Lifecycle MVP

Lifecycle User Note описывает пользовательское состояние заметки внутри плана действий.

Минимальные состояния MVP:

- создана;
- изменена пользователем;
- удалена или скрыта пользователем.

Создание User Note является пользовательским действием.

Изменение User Note, если оно допускается в MVP, является пользовательским действием.

Удаление или скрытие User Note, если оно допускается в MVP, является пользовательским действием.

Создание User Note всегда создаёт событие истории.

Редактирование User Note создаёт событие истории.

Удаление или скрытие User Note создаёт событие истории.

Администратор контента не переводит User Note между состояниями.

User Note не должна изменяться автоматически.

User Note не должна удаляться автоматически при обновлении сценария, шага, источника или Template Open Question.

Способ технической реализации удаления или скрытия User Note в этой спецификации не выбирается.

## Правила редактирования и удаления

В MVP User Note редактируется только пользователем, который владеет планом действий.

Администратор контента не редактирует User Note.

Администратор контента не удаляет User Note как часть обновления контентной модели.

Администратор контента не скрывает User Note пользователя.

Редактирование User Note не должно:

- менять событие истории, к которому была привязана заметка;
- менять текущий прогресс;
- закрывать открытый вопрос;
- менять источник;
- менять документ или требование к данным;
- менять сценарий или версию сценария;
- создавать официальный факт.

Удаление или скрытие User Note не должно:

- удалять само событие истории;
- менять текущий прогресс;
- удалять открытый вопрос;
- удалять отметку источника;
- переписывать историю плана задним числом;
- нарушать границы справочной помощи.

Перед technical specs нужно отдельно уточнить, является ли удаление физическим удалением, скрытием, архивированием или пользовательской отметкой удаления. Эта спецификация не выбирает способ реализации.

## Персональные данные и privacy-ограничения

User Note создаёт риск хранения лишних пользовательских данных, потому что пользователь может ввести произвольный текст.

Privacy-границы MVP:

- User Note должна быть короткой записью для восстановления контекста события;
- User Note не должна становиться хранилищем документов, файлов, переписки или заполненных персональных значений;
- User Note не должна парситься для принятия решений;
- User Note не должна использоваться для профилирования пользователя;
- User Note не должна использоваться для автоматического закрытия открытых вопросов;
- User Note не должна использоваться для автоматического изменения прогресса;
- User Note не должна использоваться как источник истины;
- User Note не должна становиться входом для юридических, налоговых, медицинских или финансовых выводов.

Через User Note пользователь может случайно ввести:

- адрес проживания;
- дату рождения;
- номер документа;
- номер заявления;
- номер страхования;
- сведения о работе;
- сведения о доходе;
- налоговые сведения;
- банковские реквизиты;
- сведения о гражданстве или статусе пребывания;
- семейные сведения;
- сведения о здоровье;
- данные детей, партнёра или других людей;
- содержание переписки с органом, учреждением, работодателем, страховой организацией или специалистом.

MVP должен снижать риск хранения таких данных через ограничения содержания, предупреждения и продуктовую модель, в которой заметка не является местом для заполненных персональных значений.

Дополнительная обработка заметок, поиск по содержанию, индексация, анализ, классификация, профилирование, экспорт, совместный доступ или использование заметок в автоматике требуют отдельной спецификации, оценки privacy-рисков, активного плана и обновления changelog.

## Влияние обновления сценария на заметки

User Note создаётся в контексте конкретного плана действий и версии сценария, на основе которой план был создан.

Обновление сценария администратором контента не должно автоматически менять User Notes активного плана.

При обновлении сценария:

- User Notes не редактируются администратором;
- User Notes не удаляются автоматически;
- User Notes не переносятся на новую версию сценария автоматически;
- смысл уже созданных заметок должен сохраняться в контексте версии плана;
- история активного плана не переписывается;
- пользовательский прогресс не сбрасывается;
- User Open Questions не удаляются автоматически.

Если будущая версия продукта будет переносить активный план на новую версию сценария, влияние на User Notes должно быть описано отдельным продуктовым решением до technical specs.

## Роль администратора контента

Администратор контента поддерживает продуктовую контентную модель: жизненные ситуации, сценарии, шаги, описания документов и данных, источники, ограничения, предупреждения и Template Open Questions.

Администратор контента не управляет User Notes.

Администратор контента не может:

- создавать User Notes от имени пользователя;
- редактировать User Notes;
- удалять User Notes;
- скрывать User Notes пользователя;
- использовать User Notes как источник информации;
- использовать User Notes как основание для изменения сценария без отдельного процесса анализа;
- превращать User Notes в контентную модель;
- переносить User Notes между версиями сценария без отдельного продуктового решения;
- использовать User Notes для консультации, профилирования или автоматического принятия решений.

Администратор может поддерживать предупреждения и содержание, которые помогают пользователю понимать границы заметок, но не редактирует личные заметки пользователя.

## Риски

Основные риски функции:

- User Note может превратиться в произвольный блокнот или дневник;
- User Note может начать хранить лишние персональные данные;
- User Note может быть смешана с источником информации;
- User Note может быть смешана с открытым вопросом;
- User Note может быть смешана с событием истории;
- User Note может быть воспринята как официальный факт;
- User Note может начать хранить официальную переписку или внешние ответы;
- User Note может стать обходом запрета на хранение пользовательских файлов и заполненных персональных значений;
- User Note может использоваться для автоматического изменения прогресса;
- User Note может использоваться для автоматического закрытия открытых вопросов;
- User Note может использоваться для профилирования пользователя;
- администратор контента может случайно начать редактировать личные заметки как часть контентной модели;
- обновление сценария может изменить смысл заметок в активных планах.

Для снижения рисков User Note должна оставаться короткой пользовательской записью, привязанной к событию истории внутри плана действий.

## Ограничения и предупреждения

User Note должна сохранять границу между пользовательским контекстом и внешней ответственностью.

Ограничения и предупреждения должны указывать:

- что User Note является личной пользовательской записью внутри плана действий;
- что User Note не является источником информации;
- что User Note не является официальным фактом;
- что User Note не является документом;
- что User Note не является открытым вопросом;
- что User Note не является событием истории сама по себе;
- что User Note не является консультацией;
- что User Note не подтверждает правильность документов, требований или результата внешнего процесса;
- что User Note не предназначена для хранения файлов, вложений, форм, переписки или структурированных персональных значений;
- что Nova Agent не использует User Notes для автоматических решений, профилирования или закрытия вопросов;
- что внешний ответ, учреждение или специалист остаются вне продукта.

## Приёмочные критерии

Функция "Пользовательские заметки" готова к передаче в technical specs, если:

- User Note определена как короткая пользовательская запись для восстановления контекста прохождения утверждённого сценария;
- User Note принадлежит конкретному пользователю;
- User Note принадлежит конкретному плану действий;
- User Note не существует без плана действий;
- User Note создаётся только пользовательским действием;
- добавление User Note определено как stateful-действие;
- добавление User Note может создать план действий, если пользователь до этого только просматривал сценарий;
- User Note всегда связана с событием истории;
- User Note может быть связана с шагом, открытым вопросом, отметкой источника или изменением статуса;
- User Note отделена от источника информации;
- User Note отделена от документа;
- User Note отделена от открытого вопроса;
- User Note отделена от события истории;
- User Note не является официальным фактом;
- User Note не является консультацией;
- User Note не является блокнотом или дневником;
- User Note не редактируется администратором контента;
- User Note не используется для автоматических решений;
- User Note не используется для профилирования;
- User Note не используется для автоматического закрытия вопросов;
- User Note не хранит файлы, вложения, заполненные формы, официальную переписку или структурированные персональные значения;
- privacy-границы описаны;
- риск превращения заметок в произвольный блокнот описан;
- риск смешения с Open Question, Source и History Event описан;
- влияние обновления сценария на заметки описано;
- функция не описывает базу данных, API, UI, инфраструктуру или код;
- функция соответствует бизнес-контексту, глобальной спецификации, карте функций, пользовательским сценариям, roadmap, технической архитектуре и specs 07-10.

## Открытые вопросы перед technical specs

Перед technical specs нужно уточнить:

- какой максимальный объём User Note допустим в MVP на продуктовом уровне;
- может ли пользователь редактировать User Note после создания;
- как пользователь может удалить или скрыть User Note без потери события истории;
- какие предупреждения нужны рядом с созданием заметки, чтобы снизить ввод лишних персональных данных;
- какие типы событий истории могут иметь User Note в MVP;
- может ли одно событие истории иметь несколько User Notes или только одну;
- как отделять короткую заметку к событию от произвольной дневниковой записи;
- какие действия с User Note требуют обновления roadmap или отдельной спецификации;
- какие правила хранения, удаления, экспорта или переноса User Notes нужны до реализации пользовательских данных;
- какие privacy-ограничения должны быть уточнены перед любым поиском, индексированием, анализом или совместным доступом к заметкам.
