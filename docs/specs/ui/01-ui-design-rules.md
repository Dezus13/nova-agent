# UI Spec 01: UI Design Rules

## Назначение документа

Документ фиксирует архитектурные правила интерфейса Nova Agent.

UI-01 — это не дизайн-система. Документ не выбирает цвета, шрифты, пиксели, компоненты, CSS, анимации, Figma-аннотации или frontend-фреймворки. Все визуальные решения остаются за пределами этого документа.

UI-01 фиксирует:

- какие сущности видимы какой роли и в каком состоянии;
- как каждая сущность должна быть представлена концептуально;
- как UI обязан передавать границы продукта (справочная помощь, не консультация, не официальное действие);
- какие операции разрешены на каждом концептуальном экране;
- какие UI-паттерны запрещены архитектурно;
- инварианты, которые обязательны для последующей реализации.

UI-01 служит входом для TS08: API должен знать, что UI будет показывать, что скрывать и какие payload-поля необходимы для передачи границ продукта.

## Часть I. Роли и доступ

### Роли MVP

В MVP два типа пользователей системы:

- **User** — человек, который использует Nova Agent для понимания своей жизненной или административной ситуации в Австрии.
- **Content Admin** — человек, который управляет продуктовым содержанием: сценариями, шагами, документами, источниками, предупреждениями.

Роли определены в TS04. UI-01 описывает только то, что видит и что делает каждая роль в интерфейсе.

Не входят в MVP: SuperAdmin, Moderator, Guest с ограниченным доступом к чужим данным, корпоративный кабинет, совместная работа.

### User: что видит

**Content Model (только published-состояние):**

- Life Situations — все опубликованные;
- Scenario Versions с `publication_state = published`;
- Scenario Versions с `publication_state = deprecated` или `superseded` — только если у пользователя есть Action Plan (active или completed), связанный с этой версией (Pattern B, TS07 Решение 8);
- Versioned Step Context — шаги внутри доступной Scenario Version;
- Document Requirements и Data Requirements — как описания требований, не как пользовательские файлы;
- Source Revisions — с явным типом источника (официальный / справочный / направление внешней проверки);
- Template Open Questions — как вопросы сценария, не как пользовательские вопросы;
- Warnings, Restrictions, Applicability Conditions.

**User-Owned Data (только свои):**

- Собственные Action Plans (active и completed);
- Собственные Step Progress записи;
- Собственные History Events;
- Собственные User Open Questions;
- Собственные User Notes;
- Собственные Checked Source Marks.

### User: что не видит

| Сущность | Причина |
|---|---|
| Scenario Versions с `publication_state = draft` | Только Content Admin |
| Deprecated/superseded Scenario Versions без Action Plan | Pattern B требует наличие плана (TS07 Решение 8) |
| Action Plans, Progress, History, Notes, UOQ, CSM других пользователей | Privacy isolation (TS04) |
| Content Admin operations (кто создал/обновил контент) | Не пользовательская область |
| Внутренние технические идентификаторы (raw ID) | UI показывает имена сущностей, не FK |
| `user_note_created` как context event для User Note | Это аудит-событие; FK Note → context event, не → `user_note_created` (TS07) |

### Content Admin: что видит

**Content Model (полный доступ на запись):**

- Life Situations — все состояния;
- Scenarios — список всех сценариев;
- Scenario Versions — все publication_states: draft, published, deprecated, superseded;
- Versioned Step Context — draft и published;
- Document Requirements, Data Requirements;
- Source Revisions — все;
- Template Open Questions — все;
- Warnings, Restrictions, Applicability Conditions.

Content Admin видит версионную историю сценариев, черновики и устаревшие версии — то, чего User никогда не видит.

### Content Admin: что не видит

Content Admin не имеет доступа к User-Owned Data ни одного пользователя.

| Сущность | Причина |
|---|---|
| Action Plans пользователей | User-Owned Data (TS04 Инвариант 19, TS07 Инвариант 12) |
| Step Progress | User-Owned Data |
| History Events | User-Owned Data |
| User Open Questions | User-Owned Data |
| User Notes | User-Owned Data |
| Checked Source Marks | User-Owned Data |
| PII пользователей | Privacy isolation |

Нет экрана, который показывает Content Admin план, историю или заметки конкретного пользователя. Никогда.

## Часть II. Принципы интерфейса

### Принцип 1. Контекст важнее действия

Для каждого пользовательского сценария сначала должны быть видны:

- цель сценария или шага;
- условия применимости (Applicability Conditions);
- ограничения (Restrictions);
- предупреждения (Warnings);
- источники для проверки;
- вопросы для внешней проверки (Template Open Questions).

И только затем — действия пользователя (изменить статус, добавить вопрос, добавить заметку).

Пользователь не должен совершать действие, не увидев предупреждение или ограничение, которое к нему относится. Если Warning или Restriction присутствуют в Scenario Version, они отображаются до списка шагов и до CTA «Начать план».

### Принцип 2. Nova Agent — не Task Manager

Nova Agent помогает пользователю понять путь, документы и шаги. Он не является системой управления задачами.

**Запрещено вводить в UI:**

- приоритеты (высокий / средний / низкий);
- исполнителей;
- дедлайны и SLA;
- спринты и итерации;
- канбан-доски;
- метки «Просрочено», «Срочно», «Назначено»;
- подзадачи внутри шагов;
- workflow автоматизации.

Статусы Progress существуют, но они являются пользовательскими отметками состояния прохождения, а не task-management метками.

UI не должен визуально напоминать Jira, Trello, Asana, ClickUp или Monday. Интерфейс Nova Agent — это навигационный и справочный инструмент, а не планировщик задач.

### Принцип 3. UI обязан отражать реальные границы продукта

Пользователь должен ясно понимать из UI:

- **Progress** = его личная отметка о состоянии шага. Не подтверждение от органа, учреждения или специалиста.
- **User Open Question** = его личный вопрос для внешней проверки. Не ответ, не консультация.
- **User Note** = его личная заметка для восстановления контекста. Не источник информации, не официальная запись.
- **Checked Source Mark** = его личная отметка о проверке источника. Не подтверждение актуальности источника от Nova Agent.
- **History Event** = внутреннее событие Nova Agent внутри плана. Не официальный журнал взаимодействия с органами.
- Nova Agent не является государственным органом.
- Nova Agent не является специалистом (юрист, налоговый консультант, врач).
- Nova Agent не подтверждает документы, заявления или решения.
- Nova Agent не принимает решения за пользователя.

Эти границы должны быть видны в UI не только в "справочном" разделе — они должны присутствовать рядом с каждой операцией, которая может создать ложное ощущение официального статуса.

### Принцип 4. Строгое разделение сущностей

Следующие пары сущностей должны быть явно разделены в UI:

| Пара | Что различает |
|---|---|
| Scenario vs. Action Plan | Scenario — контентная структура для всех; Action Plan — пользовательский экземпляр |
| Template Open Question vs. User Open Question | Template — часть сценария; User — личный вопрос пользователя внутри плана |
| Source vs. Checked Source Mark | Source — контентная сущность; Mark — пользовательская отметка |
| Content Model vs. User-Owned Data | Content — общий контент; User Data — только для конкретного пользователя |
| User Context vs. Content Admin Context | Разные экраны, разные операции, полная изоляция |

## Часть III. Концептуальные экраны пользователя

### U-01: Каталог жизненных ситуаций

**Назначение:** Помочь пользователю выбрать жизненную ситуацию, которая соответствует его контексту.

**Что показывается:**
- Список Life Situations с кратким описанием;
- Условия входа (какой контекст применим);
- Указание на число связанных сценариев (опционально).

**Что не показывается:**
- Сценарии без жизненной ситуации;
- Draft Scenario Versions;
- Пользовательские данные.

**Действия пользователя:** выбор жизненной ситуации → переход к U-02.

---

### U-02: Карточка жизненной ситуации

**Назначение:** Показать описание ситуации и связанные сценарии.

**Что показывается:**
- Название и описание Life Situation;
- Условия применимости (Applicability Conditions);
- Ограничения (Restrictions);
- Список связанных Scenarios (имена, краткие описания).

**Что не показывается:**
- Детали Scenario Version;
- Пользовательские данные.

**Действия пользователя:** выбор сценария → переход к U-03 или U-05 (если активный план уже есть).

---

### U-03: Просмотр сценария (без плана — read-only)

**Назначение:** Дать пользователю возможность изучить сценарий до начала прохождения.

**Что показывается (строго в этом порядке):**
1. Warnings и Restrictions — до списка шагов;
2. Цель сценария;
3. Условия применимости;
4. Template Open Questions — вопросы, которые нужно проверить до начала;
5. Список шагов с кратким описанием (read-only);
6. Связанные документы и источники на уровне сценария;
7. CTA «Начать план» (если нет активного плана).

**Важные правила:**
- Просмотр сценария не создаёт Action Plan (TS07, F09). Пользователь находится в режиме чтения.
- Если у пользователя уже есть active Action Plan по этому сценарию — вместо CTA «Начать план» показывается «Продолжить план» (переход к U-05).
- Если пользователь только смотрит без плана — Progress, History, User Notes, User UOQ не создаются.

**Что не показывается:**
- Draft Scenario Versions;
- User-Owned Data;
- Stateful-контролы (изменение статуса, создание заметки).

**Действия пользователя:** изучение → CTA «Начать план» (явное action, создающее plan).

---

### U-04: Мои планы действий

**Назначение:** Показать список всех активных и завершённых планов пользователя.

**Что показывается:**
- Список Action Plans: имя сценария (stable identity), статус (active / completed);
- Дата создания (из History Event `action_plan_created`);
- Краткий индикатор прогресса (число шагов, выполненных шагов).

**Правила:**
- Показываются только собственные планы пользователя;
- Completed план доступен только для чтения (но History, Notes, UOQ сохраняются);
- Если планов нет — пустое состояние с направлением к U-01.

**Действия пользователя:** выбор плана → переход к U-05.

---

### U-05: Детальный просмотр плана

**Назначение:** Центральный рабочий экран пользователя. Показывает текущее состояние прохождения сценария.

**Что показывается (строго в этом порядке):**
1. Имя сценария + индикатор версии (если Pattern B — version warning, см. U-09);
2. Статус плана (active / completed);
3. Disclaimer: «Nova Agent — справочная помощь. Продукт не заменяет органы, учреждения или специалистов.»;
4. Список шагов с текущим статусом Progress каждого шага;
5. Краткая сводка открытых вопросов пользователя;
6. Ссылка на историю плана (U-07);
7. Ссылка на все открытые вопросы плана (U-08).

**Правила представления Progress:**
- Каждый статус сопровождается маркировкой «Ваша отметка»;
- Изменение статуса — только явное действие пользователя; нет автоматических переходов;
- Запрещено визуально отображать переход в `not_started` (кроме начального состояния).

**Completed план:**
- Progress-статусы read-only (нельзя изменять);
- History, Notes, UOQ доступны только для чтения (изменяемость — открытый вопрос до TS08).

**Действия пользователя (в active плане):** открыть шаг (U-06), добавить UOQ (U-08), открыть историю (U-07).

---

### U-06: Детальный просмотр шага внутри плана

**Назначение:** Показать содержание шага и текущее состояние пользователя по этому шагу.

**Что показывается (строго в этом порядке):**
1. Warnings и Restrictions, связанные с этим шагом — первыми;
2. Назначение шага (зачем он нужен);
3. Входные данные: что нужно подготовить или понять;
4. Applicability Conditions (если есть);
5. Template Open Questions для этого шага;
6. Связанные Document Requirements и Data Requirements;
7. Связанные Source Revisions с типом источника;
8. Текущий статус Progress для этого шага (с маркировкой «Ваша отметка»);
9. Контролы изменения статуса Progress;
10. User Open Questions пользователя, связанные с этим шагом;
11. User Notes пользователя, связанные с событиями этого шага;
12. Checked Source Marks пользователя по источникам этого шага.

**Правила:**
- Содержание шага (Warnings, Template OQ, документы, источники) показывается до контролов статуса;
- Template OQ и User OQ визуально разделены (см. Правило разделения OQ);
- Checked Source Mark показывается рядом с Source как «Вы проверили [дата]», не как «Подтверждено»;
- Marking Source как проверенного не создаёт Action Plan (TS07 Решение 1) — это не stateful plan-creating action.

**Действия пользователя:** изменить статус Progress, создать User OQ, создать User Note, отметить Source как проверенный.

---

### U-07: История плана

**Назначение:** Восстановить контекст прохождения через хронологический журнал событий.

**Что показывается:**
- Хронологический список History Events в порядке создания (append-only);
- Для каждого события: тип события (человекочитаемый), контекст (какой шаг, какой вопрос, какой источник), временная метка;
- User Note, привязанная к событию — рядом с событием.

**Обязательный disclaimer раздела:** «История — внутренние события Nova Agent. Не является официальным журналом взаимодействия с органами, учреждениями или специалистами.»

**Правила:**
- История append-only; события не удаляются и не редактируются;
- Если User Note к событию скрыта или удалена — решение об отображении принимает TS08 (open question);
- История принадлежит плану; нет истории вне плана.

**Действия пользователя:** просмотр; управление User Note (редактировать, скрыть, удалить) в контексте события.

---

### U-08: Открытые вопросы плана

**Назначение:** Центральный список всех User Open Questions пользователя в рамках плана.

**Что показывается:**
- Список User Open Questions с текущим статусом;
- Статус из фиксированного набора: `открыт`, `требует проверки`, `ожидает внешнего ответа`, `выяснен пользователем`, `неактуален`;
- Связь с шагом или источником (если указана пользователем);
- Дата создания (из History Event).

**Правила:**
- User OQ маркированы «Ваш вопрос»;
- «Выяснен пользователем» сопровождается маркировкой «Ваша личная отметка — не профессиональное решение»;
- Изменение статуса — только явное действие пользователя; нет автоматических переходов;
- `irrelevant` (`неактуален`) — финальное состояние; обратный переход недоступен (TS07 Решение 3);
- Нет автоматического закрытия UOQ от изменения Progress.

**Действия пользователя:** создать UOQ, изменить статус UOQ, редактировать формулировку, отметить как неактуальный.

---

### U-09: Просмотр устаревшего сценария (Pattern B)

**Назначение:** Показать Scenario Version с `publication_state = deprecated` или `superseded` пользователю, у которого есть связанный Action Plan.

**Применяется только при:** наличии у пользователя Action Plan (active или completed) для этой конкретной Scenario Version.

**Что показывается дополнительно к U-03/U-05:**
- **Version warning баннер** (обязательный): «Это содержание соответствует версии сценария, с которой вы начали план. Возможно, требования изменились. Проверяйте актуальную информацию в официальных источниках.»
- Баннер не блокирует работу с планом, но должен быть заметным и постоянным в рамках этого экрана.

**Правила:**
- Пользователь без Action Plan к этой версии не видит deprecated/superseded Scenario Version — никогда;
- Баннер не удаляется действием пользователя; он остаётся на всё время работы с планом на устаревшей версии.

## Часть IV. Концептуальные экраны Content Admin

### A-01: Управление жизненными ситуациями

**Что показывается:**
- Список Life Situations (все состояния);
- Связи со Scenarios.

**Действия:** создать, редактировать, связать со Scenario.

---

### A-02: Управление сценариями

**Что показывается:**
- Список всех Scenarios (stable identities);
- Текущее publication_state последней версии;
- Число активных Scenario Versions.

**Действия:** создать новый сценарий, перейти к редактору черновика (A-03), перейти к управлению версиями (A-04).

---

### A-03: Редактор черновика сценария

**Что показывается:**
- Черновик Scenario Version (`publication_state = draft`);
- Шаги, Document Requirements, Data Requirements, Sources, Template OQ, Warnings, Restrictions, Applicability Conditions.

**Действия:** создать/редактировать/упорядочить шаги, управлять зависимостями шагов, добавить Template OQ, добавить Warnings/Restrictions, опубликовать (→ published).

**Правила:**
- Draft никогда не виден пользователю;
- Публикация фиксирует версию как immutable.

---

### A-04: Управление версиями сценария

**Что показывается:**
- Хронологический список Scenario Versions для одного Scenario;
- publication_state каждой версии;
- Дата публикации.

**Действия:** просмотреть версию, deprecate, supersede.

**Правила:**
- Deprecate/supersede не удаляет версию (исторический контекст сохраняется для Pattern B);
- Content Admin не видит, сколько пользователей имеет планы на каждой версии.

---

### A-05: Управление источниками

**Что показывается:**
- Source Revisions с типом источника (официальный / справочный / направление внешней проверки);
- Связь с Scenario Versions, Steps, Document Requirements.

**Действия:** создать Source Revision, обновить, изменить тип источника, связать с контентом.

---

### A-06: Управление Template Open Questions

**Что показывается:**
- Список Template OQ;
- Связь с Scenario, Step, Document/Data Requirement, Source.

**Действия:** создать, редактировать, связать с контентом, отметить как неактуальный.

**Правила:**
- Template OQ ≠ User OQ; Content Admin не видит и не управляет User OQ пользователей;
- Content Admin не формулирует ответ — только вопрос для внешней проверки.

---

### A-07: Управление предупреждениями и ограничениями

**Что показывается:**
- Warnings — контекст и текст;
- Restrictions — контекст применимости;
- Applicability Conditions — условия, при которых сценарий/шаг применим.

**Действия:** создать, редактировать, привязать к Scenario Version или Step.

## Часть V. Правила представления сущностей

### Scenario и Action Plan

**Scenario (read-only для пользователя):**
- Отображается как справочная структура;
- Заголовок содержит имя сценария (stable identity, не version ID);
- Warnings и Restrictions — до списка шагов;
- CTA «Начать план» — явное действие, создающее план;
- Без плана — только режим чтения.

**Action Plan:**
- Отображается как «Ваш план»;
- Имя сценария + индикатор версии (если Pattern B — version warning баннер);
- Статус: `active` или `completed`;
- При completed: контролы Progress недоступны (read-only).

**Разделение в UI:**
- «Содержание сценария» (контент, Template OQ, источники) визуально отделено от «Вашего прогресса» (Progress, User UOQ, User Notes, History).
- Пользователь не должен путать контентный шаг с пользовательской отметкой.

---

### Template Open Question и User Open Question

**Template Open Question:**
- Часть контентной модели; отображается в составе сценария или шага;
- Маркировка: «Вопросы для проверки» (как часть справочного содержания);
- Не имеет пользовательского статуса; пользователь не может его изменить;
- Показывается в U-03 (просмотр сценария без плана) и в U-06 (шаг внутри плана).

**User Open Question:**
- Пользовательский вопрос; принадлежит конкретному плану;
- Маркировка: «Ваш вопрос»;
- Имеет пользовательские статусы и переходы;
- Создаётся только явным действием пользователя внутри плана;
- «Выяснен пользователем» маркируется: «Ваша личная отметка — не профессиональное решение».

**Правило разграничения:**
Template OQ и User OQ никогда не смешиваются в одном списке без явной визуальной и семантической метки владения.

---

### Source и Checked Source Mark

**Source (Source Revision):**
- Тип источника указывается всегда: «Официальный источник» / «Справочный материал» / «Направление внешней проверки: орган / учреждение / специалист»;
- «Проверьте актуальность» — показывается рядом с источником, особенно если требования зависят от города, учреждения, даты или статуса пользователя;
- Source не становится «проверено Nova Agent» — никогда.

**Checked Source Mark:**
- Показывается рядом с Source внутри плана как «Вы отметили этот источник [дата]»;
- Не превращает Source в «подтверждённый источник»;
- Отметка источника как проверенного НЕ создаёт Action Plan (TS07 Решение 1 — это не plan-creating action);
- Если Plan не существует, контрол «Отметить как проверенный» недоступен.

---

### Step Progress

**Представление статусов:**

| Статус (internal) | Отображение пользователю |
|---|---|
| `not_started` | «Не начато» |
| `in_progress` | «В процессе» |
| `awaiting_external_response` | «Ожидает внешнего ответа» |
| `completed` | «Выполнено» |
| `requires_check` | «Требует проверки» |

**Обязательные правила:**
- Каждый статус сопровождается: «Ваша отметка»;
- «Выполнено» не означает «официально принято»; рядом должно быть явно: «Ваша отметка о завершении шага»;
- «Ожидает внешнего ответа» — пользовательская отметка ожидания, не системное отслеживание;
- Изменение статуса — только явное действие пользователя (нет автоматических переходов);
- Контрол перехода в `not_started` скрыт (нет возможности вернуться в `not_started` из любого другого статуса — TS07);
- При completed плане — все контролы изменения Progress недоступны.

---

### History Events

**Представление:**
- Хронологический список (append-only; от старых к новым или наоборот — решение реализации);
- Тип события отображается человекочитаемо:

| event_type | Отображение |
|---|---|
| `action_plan_created` | «Начало плана» |
| `action_plan_completed` | «Завершение плана» |
| `progress_status_changed` | «Изменение статуса шага» |
| `source_checked` | «Отметка источника» |
| `user_open_question_created` | «Добавлен вопрос» |
| `user_open_question_status_changed` | «Изменён статус вопроса» |
| `user_open_question_edited` | «Вопрос отредактирован» |
| `user_note_created` | «Добавлена заметка» |
| `user_note_edited` | «Заметка отредактирована» |
| `user_note_hidden` | «Заметка скрыта» |
| `user_note_deleted` | «Заметка удалена» |

**Обязательный disclaimer:** «История — внутренние события Nova Agent. Не является официальным журналом взаимодействия с органами, учреждениями или специалистами.»

**User Note в контексте события:**
- User Note показывается рядом с тем History Event, на который ссылается её FK (context event);
- Аудит-событие `user_note_created` показывается как «Добавлена заметка», а не как context для Note;
- Одно History Event может иметь несколько User Notes (cardinality N:1, TS05/TS07).

---

### User Note

**Представление:**
- Маркировка: «Ваша заметка»;
- Контекст: «К событию: [человекочитаемый тип context event]»;
- Действия: редактировать, скрыть, удалить — только в active плане (изменяемость в completed плане — open question, TS08);
- User Note — не источник информации, не Open Question, не официальная запись;
- Создание User Note создаёт Action Plan, если план не существует (stateful action, TS07 Решение 2).

---

### Warning, Restriction, Applicability Condition

**Warning:**
- Отображается первым — до любого контента шага или сценария;
- Формулировка начинается с «Предупреждение:»;
- Не скрывается действием пользователя в текущей сессии;
- Не означает «Nova Agent запрещает» — информирует о зоне внешней ответственности.

**Restriction:**
- Отображается рядом с контентом, к которому применяется;
- Формулировка начинается с «Ограничение:»;
- Не является автоматическим выводом о неприменимости шага для пользователя.

**Applicability Condition:**
- Отображается до начала сценария или до шага, к которому применяется;
- Формулировка начинается с «Условие применимости:»;
- Пользователь самостоятельно решает, применимо ли условие к его ситуации; Nova Agent не принимает это решение.

## Часть VI. Правила передачи границ продукта в UI

### Обязательный boundary disclaimer

На каждом экране, где пользователь работает с Action Plan (U-05, U-06, U-07, U-08, U-09), присутствует постоянный элемент:

«Nova Agent — справочная и организационная помощь. Продукт не является государственным органом, специалистом или консультантом. Все отметки прогресса, вопросы и заметки отражают только ваши собственные записи. Nova Agent не подтверждает документы, сроки, применимость требований или результаты официального процесса.»

Этот элемент:
- присутствует на всех plan-экранах постоянно, а не только при первом входе;
- не скрывается навсегда пользовательским действием (конкретный UX — решение реализации, но исчезновение навсегда — запрещено).

### Обязательные лейблы для пользовательских отметок

| Сущность | Обязательная маркировка |
|---|---|
| Progress статус | «Ваша отметка» рядом со статусом |
| User Open Question | «Ваш вопрос для внешней проверки» |
| UOQ статус «Выяснен пользователем» | «Ваша личная отметка — не профессиональное решение» |
| User Note | «Ваша заметка» |
| Checked Source Mark | «Вы отметили этот источник [дата]» |
| History | «Внутренние события Nova Agent» |
| Завершение плана | «Ваша отметка о завершении» |

### Запрещённые формулировки в UI

Следующие формулировки **никогда** не должны появляться в интерфейсе Nova Agent:

- «Принято», «Одобрено», «Принято учреждением»;
- «Документы проверены», «Документы в порядке», «Документы достаточны»;
- «Подано», «Зарегистрировано», «Получено»;
- «Вы имеете право», «Право подтверждено»;
- «Готово к подаче», «Можно не проверять»;
- «Проверено Nova Agent», «Подтверждено Nova Agent»;
- «Решение получено», «Результат получен»;
- «Автоматически определено», «Автоматически завершено».

Эти формулировки создают ложное ощущение официального статуса, что противоречит глобальной спецификации и функциональным specs.

## Часть VII. UI-инварианты

1. **Role Separation.** Пользовательские экраны (U-01–U-09) и admin-экраны (A-01–A-07) полностью изолированы. Нет экрана, доступного обеим ролям одновременно.

2. **Privacy Isolation.** Пользователь видит только собственные Action Plans, Progress, History Events, User Open Questions, User Notes, Checked Source Marks. Данные других пользователей недоступны никогда.

3. **Draft Hidden.** Scenario Versions с `publication_state = draft` никогда не отображаются пользователю ни на каком экране.

4. **Pattern B Scope.** Deprecated или superseded Scenario Version показывается пользователю только при наличии у него Action Plan (active или completed), связанного с этой версией. Без плана — доступ закрыт.

5. **Version Warning Required.** При Pattern B на всех plan-экранах присутствует постоянный version warning баннер о том, что содержание соответствует версии момента создания плана.

6. **User Mark Labeling.** Каждый Progress-статус, каждый User OQ статус, каждый Checked Source Mark сопровождается явной маркировкой «Ваша отметка» или аналогичной фразой. Маркировка обязательна и не может быть убрана.

7. **History Disclaimer.** История плана на экране U-07 всегда содержит disclaimer о том, что это внутренние события Nova Agent, а не официальный журнал.

8. **Source Type Label.** Каждый Source (Source Revision) отображается с явным типом: официальный / справочный / направление внешней проверки. Нет Source без типа.

9. **No Automatic Transitions.** Никакой статус Progress и никакой статус User OQ не изменяется автоматически — только явным действием пользователя.

10. **Context Before Action.** На U-03 и U-06 Warnings и Restrictions отображаются до CTA и до контролов действий. Пользователь не может совершить действие, не увидев предупреждение.

11. **Viewing ≠ Starting Plan.** Просмотр сценария на U-03 не создаёт Action Plan. План создаётся только явным stateful-действием.

12. **Checked Source ≠ Endorsement.** Checked Source Mark показывается как «Вы отметили», а не как «Подтверждено». Отметка не меняет тип источника и не превращает его в официальный факт.

13. **UOQ ≠ Answer.** User Open Question — вопрос, а не ответ. Статус «Выяснен пользователем» — личная отметка пользователя, не профессиональное решение. UI маркирует это явно.

14. **Admin No User Data.** Ни один admin-экран не отображает User-Owned Data ни одного пользователя. Нет административного просмотра планов, истории или заметок.

15. **Template OQ vs. User OQ Separation.** Template Open Questions и User Open Questions никогда не смешиваются в одном списке без явной семантической метки владения.

16. **Boundary Banner Required.** На всех plan-экранах (U-05, U-06, U-07, U-08, U-09) постоянно присутствует boundary disclaimer. Disclaimer не исчезает навсегда.

17. **No Task Manager Elements.** Никакого приоритета, дедлайна, исполнителя, спринта, канбан-доски или метки «Просрочено» в пользовательском интерфейсе.

18. **Completed Plan Read-Only Progress.** В completed Action Plan контролы изменения Progress недоступны. Данные доступны только для чтения.

19. **Source Revision Currency Warning.** Рядом с каждым Source Revision, требования которого могут зависеть от города, учреждения, даты или статуса пользователя, показывается «Проверьте актуальность требований».

20. **Not-Started Transition Hidden.** Переход Progress в `not_started` недоступен для пользователя из любого другого статуса (TS07 State Machine). Контрол перехода в `not_started` в UI отсутствует.

21. **No Plan from Source Mark.** Отметка источника как проверенного не создаёт Action Plan и не вызывает plan-creation flow (TS07 Решение 1).

22. **UOQ Irreversible Final.** После перехода UOQ в статус `irrelevant` (`неактуален`) — обратного перехода нет. Контрол обратного перехода в UI отсутствует (TS07 Решение 3).

## Часть IX. Риски

### R-01. Ложный официальный статус

Если Progress-статус «Выполнено» или «Ожидает внешнего ответа» отображается без маркировки «Ваша отметка», пользователь может воспринять его как официальное подтверждение от органа или учреждения.

Снижение: обязательная маркировка (UI-инвариант 6); запрещённые формулировки явно перечислены.

### R-02. Task-manager drift

Если в интерфейс постепенно добавляются task-manager элементы (например, «срок» или «напоминание»), продукт теряет границу между справочной помощью и управлением задачами.

Снижение: UI-инвариант 17; явный список запрещённых паттернов; проверка на соответствие каждого нового UI-элемента карте функций.

### R-03. Privacy leak

Если visibility rules нарушены — пользователь видит чужие данные, или Content Admin видит пользовательские данные.

Снижение: UI-инварианты 2 и 14; изоляция контекстов ролей; раздельные экраны без пересечения.

### R-04. Draft scenario leak

Если draft Scenario Version становится видна пользователю через API или UI-ошибку — пользователь видит неготовый контент.

Снижение: UI-инвариант 3; API должен фильтровать draft по роли (формируется требование для TS08).

### R-05. Pattern B нарушение

Если UI даёт пользователю доступ к deprecated/superseded Scenario Version без Action Plan — нарушение TS05 Amendment 03 и TS07 Решение 8.

Снижение: UI-инвариант 4; API должен возвращать флаг доступа по Pattern B (требование для TS08).

### R-06. Смешение Template OQ и User OQ

Если Template OQ и User OQ не разграничены, пользователь не понимает, что является частью общего содержания, а что — его личным вопросом.

Снижение: UI-инвариант 15; раздельные секции с явными лейблами владения.

### R-07. Источник как гарантия

Если рядом с Source не показан его тип и не показано «Проверьте актуальность», пользователь может воспринять источник как гарантию применимости требования.

Снижение: UI-инвариант 8; UI-инвариант 19; запрещённые формулировки.

### R-08. Контекст после действия

Если Warning или Restriction показываются после CTA «Начать план» или после контрола изменения статуса, пользователь совершает действие без нужной информации.

Снижение: UI-инвариант 10; явный порядок отображения в U-03 и U-06.

### R-09. User Note как источник

Если User Note показывается в списке Sources или рядом с Template OQ без явной маркировки «Ваша заметка», пользователь может воспринять её как официальный источник.

Снижение: UI-инвариант 6; маркировка «Ваша заметка» обязательна.

### R-10. Boundary disclaimer исчезает

Если boundary disclaimer показывается только один раз и закрывается навсегда — пользователь работает с планом без постоянного напоминания о границах продукта.

Снижение: UI-инвариант 16; disclaimer постоянный на всех plan-экранах.

## Часть XI. Решения до TS08

Следующие UI-решения должны быть зафиксированы до написания TS08, так как они напрямую определяют структуру API:

| Решение | Почему нужно до TS08 |
|---|---|
| Инвентарь концептуальных экранов (U-01–U-09, A-01–A-07) | API нужны эндпоинты под каждый экран |
| Visibility rules (роль → сущность) | API определяет, что возвращать и кому |
| Boundary communication vocabulary (тип источника, user mark label) | API payload должен содержать поля для disclaimer |
| Запрещённые UI-паттерны | Некоторые паттерны запрещают создавать API под них |
| Plan-creation trigger | API должен знать, когда план создаётся (и не создаётся при просмотре) |
| Pattern B signal | API должен возвращать флаг version-warning |
| Template OQ vs. User OQ differentiation | API response должен разграничить ownership |
| Checked Source Mark не является plan-creating | API не создаёт план при этом действии |
| User Note FK → context History Event | API должен возвращать context_event вместе с User Note |
| Completed Action Plan read-only Progress | API не должен принимать Progress-изменения для completed плана |
